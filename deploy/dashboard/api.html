Maybe Finance API

Overview

Design principles

All endpoints return JSON and are scoped to the authenticated user or their family. API responses are designed for direct use by automation systems, mobile clients, and tools like Siri Shortcuts.

Family-scoped access. Accounts, categories, and transactions are resolved from the authenticated user’s family only. A valid ID from another family still returns 404.
Server-side category resolution. Transaction create and update endpoints accept category_name in addition to category_id. This is useful when your client only knows the category label, not the UUID.
Authentication

Use OAuth bearer tokens or API keys

The API supports two authentication methods. If both are sent, OAuth takes precedence over API key authentication.

Method Header Notes
OAuth 2.0 Authorization: Bearer <token> Supports read and read_write scopes.
API key X-Api-Key: <key> Supports the same scope model and rate limit tracking. 
curl -H "X-Api-Key: YOUR_API_KEY" \
http://localhost:3000/api/v1/accounts

curl -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
http://localhost:3000/api/v1/transactions
Scopes are hierarchical. read_write includes read access. Write endpoints require read_write.
Rate Limits

API key requests include usage headers

API key requests are rate-limited and include rate-limit headers on successful responses. OAuth requests authenticate successfully but do not currently expose the same detailed usage metadata.

Header Description Example
X-RateLimit-Limit Total requests allowed in the current window 100
X-RateLimit-Remaining Remaining requests in the current window 99
X-RateLimit-Reset Seconds until the limit resets 3540
Retry-After Returned when the limit is exceeded 3540
Errors

Standard error shapes

Most failures use a predictable JSON shape with an error key and a human-readable message.

{
  "error": "unauthorized",
  "message": "Access token or API key is invalid, expired, or missing"
}

{
  "error": "validation_failed",
  "message": "Category could not be found",
  "errors": ["Category 'Dining' was not found"]
}
Status Error When it happens
400 bad_request Missing required parameters such as refresh_token.
401 unauthorized Missing, expired, revoked, or invalid credentials.
403 insufficient_scope The credential does not include the required scope.
404 not_found or record_not_found The resource does not exist or does not belong to the current family.
422 validation_failed The request payload is syntactically valid but fails business rules.
429 rate_limit_exceeded The API key exceeded its current rate limit window.
Pagination

List endpoints use page-based pagination

Resource list endpoints return a top-level collection and a pagination object. Most endpoints default to 25 items per page and allow up to 100. 

{
  "pagination": {
    "page": 1,
    "per_page": 25,
    "total_count": 14,
    "total_pages": 1
  }
}

Authenticate an existing user, optionally with MFA, and issue fresh OAuth credentials.

Categories

Discover category IDs and hierarchy

Categories are now exposed over the REST API. This is especially useful for Siri Shortcuts and automations that need to turn a category label into a stable UUID.

GET/api/v1/categories

Query parameters

Name Type Description
classification string Filter to income or expense.
parent_id uuid Return only subcategories under a specific parent.
roots boolean When true, return only root-level categories.
search string Case-insensitive partial match on category name.
page integer Page number. Defaults to 1.
per_page integer Items per page. Defaults to 25, max 100. 
curl -s -H "X-Api-Key: YOUR_API_KEY" \
http://localhost:3000/api/v1/categories?roots=true | jq
GET/api/v1/categories/:id

Transactions

Create, update, query, and delete transactions

The transactions API supports filtering, full record retrieval, writes, and deletes. Transaction write endpoints accept either category_id or category_name.

Important: nature controls the sign of the amount. Use expense or outflow for money leaving the account. Use income or inflow for deposits.
GET/api/v1/transactions

Query parameters

Name Type Description
account_iduuidFilter to a single account.
account_idsarrayFilter to multiple accounts.
category_iduuidFilter to a single category.
category_idsarrayFilter to multiple categories.
merchant_iduuidFilter to a single merchant.
merchant_idsarrayFilter to multiple merchants.
start_datedateInclusive lower date bound.
end_datedateInclusive upper date bound.
min_amountdecimalInclusive minimum amount.
max_amountdecimalInclusive maximum amount.
tag_idsarrayFilter by tags.
typeenumincome or expense.
searchstringMatches name, notes, or merchant name.
pageintegerPage number.
per_pageintegerItems per page, max 100.
GET/api/v1/transactions/:id

Return a single transaction including account, category, merchant, tags, and transfer metadata.

POST/api/v1/transactions

Create a new transaction. account_id is required.

Request body

{
  "transaction": {
    "account_id": "ACCOUNT_UUID",
    "name": "Lunch",
    "amount": 18.50,
    "date": "2026-03-10",
    "currency": "USD",
    "nature": "expense",
    "category_name": "Food & Drink",
    "notes": "Lunch with Tom & Ava"
  }
}
category_name resolution: Matching is exact by category name but case-insensitive. If both category_id and category_name are supplied, category_id wins.
PUT/api/v1/transactions/:id

Update an existing transaction using the same transaction payload shape.

DELETE/api/v1/transactions/:id

Delete a transaction. Successful responses return { "message": "Transaction deleted successfully" }.

AI

Chats

AI endpoints require the authenticated user to have AI features enabled. Read operations require read access. Write operations require read_write.

GET/api/v1/chats

List chats for the current user with pagination metadata.

GET/api/v1/chats/:id

Return a chat and its messages.

POST/api/v1/chats

Create a chat. If message is provided, the API also creates the initial user message and queues an assistant response.

{
  "title": "Spending review",
  "message": "Summarize my spending this month",
  "model": "gpt-4"
}
PATCH/api/v1/chats/:id

Update a chat title.

DELETE/api/v1/chats/:id

Delete a chat. Successful responses return 204 No Content.

AI

Messages

POST/api/v1/chats/:chat_id/messages

Create a user message within a chat and enqueue an assistant response.

{
  "content": "What changed in my balances this week?",
  "model": "gpt-4"
}
POST/api/v1/chats/:chat_id/messages/retry

Retry the most recent assistant response for the chat.

Examples

Common integration flows

List categories for Siri Shortcuts

curl -s -H "X-Api-Key: YOUR_API_KEY" \
http://0.0.0.0:3000/api/v1/categories | jq

Create an expense using category_name

curl -X POST http://0.0.0.0:3000/api/v1/transactions \
  -H "X-Api-Key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "transaction": {
      "account_id": "YOUR_ACCOUNT_ID",
      "name": "Lunch",
      "amount": 18.50,
      "category_name": "Food & Drink",
      "nature": "expense",
      "date": "2026-03-10"
    }
  }' | jq

Create a deposit using category_name

curl -X POST http://0.0.0.0:3000/api/v1/transactions \
  -H "X-Api-Key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "transaction": {
      "account_id": "YOUR_ACCOUNT_ID",
      "name": "Paycheck",
      "amount": 2500.00,
      "category_name": "Income",
      "nature": "income",
      "date": "2026-03-10"
    }
  }' | jq

Notes on ampersands and spaces

No escaping is required inside JSON strings. A category such as Food & Drink can be sent exactly as written inside the JSON body. URL encoding is only required when the value is part of the URL or query string.
